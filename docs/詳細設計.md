## 2. システム構成

```
Flutter App (Web / Android / iOS)
       │
       ├── Firebase Authentication  (認証 / メール・パスワード)  ✅
       ├── Cloud Firestore          (データストア)                ✅
       ├── Firebase Storage         (画像ストレージ)              ✅
       ├── Cloud Functions          (承認フロー / 1st gen)        ✅
       │     ├── onUserCreated      (users 作成トリガー → 承認依頼メール)
       │     └── handleApproval     (HTTPS / 承認・却下リンク処理)
       ├── Trigger Email 拡張        (mail コレクション → SMTP 送信)  ✅
       └── Firebase Messaging       (プッシュ通知 / PS_07)        ❌ 未導入
```

**地図**: OpenStreetMap (flutter_map) — Google Maps API は不使用  
**Firebase プロジェクト**: `flutter-afterglow`  
**リージョン**: Firestore / Cloud Functions ともに `asia-northeast2`（大阪）。
Firestore トリガー関数はデータベースと同一リージョンに置く必要があるため統一している。  
**料金プラン**: Cloud Functions / Trigger Email 利用のため **Blaze プラン**が前提。

> Phase1 認証基盤の構成・運用手順の詳細は
> [phase1_roadmap.md](./prompts/phase1_roadmap.md) を参照。

### 4.3 管理者承認フロー（Phase1・実装済み）

ドメイン制限を設けず誰でも登録申請でき、管理者がメールのリンクで承認/却下する方式。

```
[ユーザー] RegisterPage で登録
   │ AuthService.register: Auth ユーザー作成 → users/{uid} を approved:false で作成
   ▼
[Cloud Functions] onUserCreated（users/{uid} 作成トリガー）
   │ ・registrationApprovals/{uid} に承認トークンを保存
   │ ・mail/{uid} に承認/却下リンク入りメールを書き込み
   ▼
[Trigger Email 拡張] mail コレクションを監視し管理者へ SMTP 送信
   ▼
[管理者] メール内の「承認する」/「却下する」リンクをクリック
   ▼
[Cloud Functions] handleApproval（HTTPS）
   │ ・token を registrationApprovals/{uid} と照合
   │ ・approve → users/{uid}.approved = true、承認ドキュメント削除
   │ ・reject  → Auth ユーザー・users・承認ドキュメントを削除
   ▼
[アプリ] AuthGate が approved の変化を検知 → PendingApprovalPage から MapScreen へ自動遷移
```

- **関数世代/リージョン**: 1st gen / `asia-northeast2`。`handleApproval` の URL を
  `https://asia-northeast2-flutter-afterglow.cloudfunctions.net/handleApproval` と決定的にするため 1st gen を採用。
- **管理者メール**: 既定 `yokochi5123@gmail.com`（環境変数 `ADMIN_EMAIL` で上書き可）。
- **最初の admin**: 自分が登録後、`users/{uid}.role` を手動で `'admin'` に変更する。

### 6.1 AuthService（✅ 実装済み）

```dart
class AuthService {
  AuthService({FirebaseAuth? auth, FirebaseFirestore? firestore}); // テスト用に注入可
  Stream<User?> get authStateChanges;
  User? get currentUser;
  String? get currentUserId;
```

## 7. セキュリティ設計（NFR_01）— ✅ 実装済み

### 7.1 Firebase Authentication

- メール/パスワード認証（`firebase_auth` パッケージ）
- パスワードはクライアントで送信 → Firebase Auth がハッシュ化して保存
- **承認制**: ログインできても `users/{uid}.approved == true` でなければアプリ本体（投稿の読み書き）にアクセスできない（§4.3）

### 7.2 Firestore セキュリティルール（`firestore.rules` — 実装済み）

設計方針:
- 投稿・コメント・リアクションの読み書きは **承認済み (`approved`) ユーザーのみ**（`isApproved()`）
- `users` の作成は本人のみ、かつ `approved:false` / `role:'member'` を強制
- `users` の更新で本人は `approved` / `role` を変更できない（昇格・自己承認の防止）。変更は admin のみ
- `registrationApprovals` / `mail` はクライアントから読み書き不可（Functions / 拡張専用）

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function isSignedIn() { return request.auth != null; }
    function userDoc(uid) {
      return get(/databases/$(database)/documents/users/$(uid)).data;
    }
    function isApproved() {
      return isSignedIn()
        && exists(/databases/$(database)/documents/users/$(request.auth.uid))
        && userDoc(request.auth.uid).approved == true;
    }
    function isAdmin() {
      return isSignedIn()
        && exists(/databases/$(database)/documents/users/$(request.auth.uid))
        && userDoc(request.auth.uid).role == 'admin';
    }

    match /users/{userId} {
      allow read: if isSignedIn();
      allow create: if isSignedIn()
        && request.auth.uid == userId
        && request.resource.data.approved == false
        && request.resource.data.role == 'member';
      allow update: if (isSignedIn()
          && request.auth.uid == userId
          && request.resource.data.approved == resource.data.approved
          && request.resource.data.role == resource.data.role)
        || isAdmin();
      allow delete: if isAdmin();
    }

    match /posts/{postId} {
      allow read: if isApproved();
      allow create: if isApproved()
        && request.resource.data.userId == request.auth.uid;
      allow update, delete: if isApproved()
        && (resource.data.userId == request.auth.uid || isAdmin());

      match /comments/{commentId} {
        allow read: if isApproved();
        allow create: if isApproved()
          && request.resource.data.userId == request.auth.uid;
        allow update, delete: if isApproved()
          && (resource.data.userId == request.auth.uid || isAdmin());
      }

      match /reactions/{reactionUserId} {
        allow read: if isApproved();
        allow write: if isApproved() && request.auth.uid == reactionUserId;
      }
    }

    match /registrationApprovals/{userId} { allow read, write: if false; }
    match /mail/{mailId} { allow read, write: if false; }
  }
}
```

> **未反映の将来コレクション**: `albums` / `events` / `notifications` のルールは
> 各機能の実装時に追加する（現状の `firestore.rules` には未定義）。

### 7.3 Firebase Storage セキュリティルール（`storage.rules` — 実装済み）

Storage ルールは Firestore を参照できないため `approved` 判定は行えない。
書き込みは「ログイン済み」かつ「本人ディレクトリ配下」かつ「10MB 未満の画像」に限定し、
投稿自体は Firestore ルールで承認済みユーザーに限定することで担保する。

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {

    function isSignedIn() { return request.auth != null; }
    function isValidImage() {
      return request.resource.size < 10 * 1024 * 1024
        && request.resource.contentType.matches('image/.*');
    }

    match /posts/{userId}/{allPaths=**} {
      allow read: if isSignedIn();
      allow write: if isSignedIn() && request.auth.uid == userId && isValidImage();
    }
    match /profiles/{userId}/{allPaths=**} {
      allow read: if isSignedIn();
      allow write: if isSignedIn() && request.auth.uid == userId && isValidImage();
    }
  }
}
```

## 8. パフォーマンス設計（NFR_02）

### 8.1 画像アップロード圧縮

アップロード前に `flutter_image_compress` パッケージで処理する。

| 条件 | 処理 |
|---|---|
| 短辺 > 2048px | 短辺 2048px にリサイズ（アスペクト比維持）|
| 短辺 ≤ 2048px | リサイズなし |
| 品質 | JPEG 品質 85% |
| ファイルサイズ | 圧縮後でも 10MB 超の場合はアップロード拒否 |

### 8.2 フィードページ

- Firestore クエリに `limit(20)` を設定（NFR_02 の3秒以内目標）
- スクロール末端で `startAfterDocument` による追加取得（無限スクロール）
- 画像は `CachedNetworkImage` でディスクキャッシュ

### 8.3 地図ページ

- タイルは `flutter_map` の標準キャッシュを使用（NFR_02 遅延読み込み）
- 画像プリロードは実装済み（MapScreen の `precacheImage` ループ）
